# AXIS / AXI-Lite 基础 - 数据传送带和控制台

## 核心结论

在固存样机里，**AXIS 负责搬数据，AXI-Lite 负责管系统**。

用费曼法说：

```text
AXIS     = 仓库传送带：箱子一箱一箱往前走，重点是别断流、别堵死。
AXI-Lite = 仓库控制台：设置工作模式、查看状态、触发故障注入、查询重建进度。
```

这两件事必须分开理解。否则很容易把“高速数据怎么流动”和“软件怎么配置系统”混成一团。

## 为什么技术要求同时写 AXIS 和 AXI-Lite

文档里要求：

- 数据接口：多路输入数据流，AXIS 64-bit；
- 管理接口：AXI-Lite；
- 能配置 RAID 模式、条带大小、启动/停止阵列；
- 能查询成员盘状态、错误计数、重建进度；
- 能做错误注入和异常场景测试。

翻译成学习问题就是：

| 工程问题 | 应该看哪条总线 | 新人要先懂什么 |
|---|---|---|
| 8 路载荷数据怎么持续进来 | AXIS | valid/ready、背压、帧边界、缓存 |
| RAID0/1/5 模式怎么设置 | AXI-Lite | 寄存器地址、写配置、读状态 |
| SSD 掉盘后系统怎么报告 | AXI-Lite | 状态位、中断、错误计数 |
| 重建时数据还怎么流 | AXIS + AXI-Lite | 数据面继续搬，控制面报告进度 |

## AXIS：先理解 valid/ready

AXIS 最小握手只有一个核心规则：

```text
valid = 发送方说：我这里有一个有效数据
ready = 接收方说：我现在接得住
valid && ready = 这一拍真正传输成功
```

像两个人递箱子：

```text
发送方举箱子(valid=1)
接收方伸手(ready=1)
两件事同时发生，箱子才算交接成功
```

如果接收方暂时忙：

```text
valid=1, ready=0  -> 数据必须保持住，不能偷偷丢掉
```

这就是“背压”。在 8 路输入固存里，背压很关键：某一路 SSD 或 RAID 内部暂时处理不过来时，前级不能把数据无声丢掉。

## 8 路输入为什么不是简单相加

8 路数据进入 FPGA 后，需要解决三个问题：

1. **仲裁**：这一拍先服务哪一路？轮询、优先级、按包调度都可以；
2. **缓存**：下游暂时 busy 时，数据先放哪里？FIFO/DDR/片上 RAM；
3. **统计与限流**：哪一路丢包、堵塞、超时，需要能被控制面看见。

费曼类比：

```text
8 条传送带同时送箱子到一个分拣口。
如果分拣口忙，有的箱子要排队；
如果某条带一直很急，调度员要决定先处理谁；
如果箱子堆满了，控制台必须报警。
```

所以第一阶段不应该直接写“8 路 AXIS 汇聚器”。更合适的是先画清楚数据路径和背压规则。

## AXI-Lite：控制台不是数据通道

AXI-Lite 慢一些、简单一些，适合做寄存器控制。

在本项目语境里，它最少要表达四类信息：

### 1. 配置寄存器

```text
RAID_MODE       0=RAID0, 1=RAID1, 5=RAID5
STRIPE_SIZE     条带大小
ARRAY_ENABLE    启动/停止阵列
REBUILD_ENABLE  是否允许后台重建
```

### 2. 状态寄存器

```text
ARRAY_STATE     normal / degraded / rebuilding / failed
MEMBER_STATUS   每块SSD在线、掉线、重建中、隔离
REBUILD_PROGRESS 0~100%
```

### 3. 计数与遥测

```text
INPUT_BYTES     输入字节数
WRITE_BYTES     写入SSD字节数
ERROR_COUNT     超时/校验/掉盘计数
TEMP/HEALTH     温度、健康信息摘要
```

### 4. 错误注入

```text
INJECT_DROP_DISK     模拟某块盘掉线
INJECT_TIMEOUT       模拟NVMe响应超时
INJECT_PARITY_ERROR  模拟校验不一致
```

这些寄存器不是最终工程表，而是学习草案：帮助读者理解“为什么技术要求不只要数据能写，还要状态能查、异常能测”。

## 数据面和控制面怎么配合

一个 RAID5 重建场景可以这样理解：

```text
1. 控制面发现 disk2 掉线：MEMBER_STATUS[disk2] = offline
2. 数据面进入 degraded read：读缺失块时用 XOR 重构
3. 控制面允许插入新盘并启动重建：REBUILD_ENABLE = 1
4. 数据面后台读幸存盘 + parity，写新盘
5. 控制面不断更新 REBUILD_PROGRESS
6. 完成后 ARRAY_STATE 回到 normal
```

关键点：

- 数据面保证数据继续按规则流动；
- 控制面保证系统状态可见、动作可控；
- 验证面用错误注入证明异常路径真的跑过。

## 和 NVMe Host 的关系

AXIS/AXI-Lite 是 FPGA 内部/系统侧接口学习重点；NVMe Host 是 SSD 设备侧的大模块。

三种常见路线：

| 路线 | 优点 | 风险 |
|---|---|---|
| 厂商/商业 NVMe Host IP | 工程落地最快，性能和兼容性更可控 | 成本、授权、可调试性受限 |
| SoC/CPU 协同管理 SSD | 软件生态成熟，调试方便 | 数据路径可能绕远，实时性和带宽要验证 |
| 自研/深度改造开源 Host | 可控性强，适合研究 | 工作量巨大，兼容性和稳定性风险最高 |

本仓库目前只把 NVMe Host 当成设备面边界，不把它作为第一阶段代码目标。

## 第一阶段应该做到什么

做到：

- 能解释 AXIS 和 AXI-Lite 分别解决什么问题；
- 能说出 valid/ready 为什么能表达背压；
- 能画出 8 路输入、RAID、NVMe 之间的数据关系；
- 能列出最小控制寄存器草案；
- 能说明错误注入和重建进度为什么是验收必需项。

不做：

- 不写完整 AXIS 汇聚 RTL；
- 不写完整 AXI-Lite 从设备；
- 不实现 NVMe Host；
- 不承诺 20/40 Gbps 性能。

下一步如果要做小实验，可以只做一个“2 路 AXIS valid/ready 仲裁玩具模型”，用来观察背压，而不是直接上 8 路工程实现。
