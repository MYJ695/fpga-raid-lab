# NVMe Host 实现路线 - 为什么先用“抽象磁盘端口”

## 核心结论

在这个教程里，**NVMe SSD 先不要当成第一阶段 RTL 目标**。

用费曼法说：

```text
RAID 层      = 仓库调度规则：数据分到哪块盘、校验放在哪里、坏盘怎么恢复。
NVMe Host   = 通往真实 SSD 的高速公路收费站 + 交通规则 + 排队系统。
PCIe PHY/IP = 高速公路本身。
```

如果新人一上来同时学 RAID、AXIS、AXI-Lite、PCIe、NVMe queue、DMA、板级时序，就会把问题混成一个大黑盒。

更稳的路线是：

```text
先把 RAID 行为跑通
  -> 用 virtual disk / BRAM / 文件模型当“假 SSD”
  -> 再定义抽象 disk port
  -> 最后才选择真实 NVMe 接入路线
```

## 技术要求里 NVMe 相关关键词

《固存样机技术要求》里关心的不是“会背 NVMe 名词”，而是系统最终能接真实盘、能稳定工作、能被验收。

常见关键词可以拆成：

| 关键词 | 新人解释 | 对教程的影响 |
|---|---|---|
| NVMe SSD | 真实高速固态盘 | 先作为设备面目标，不直接实现 |
| PCIe Gen3 x4 | 每块盘的高速连接通道 | 涉及板卡、IP、时序和资源 |
| NVMe 1.2/1.3 | SSD 通信协议版本 | 需要 Host 控制器理解命令/队列 |
| Namespace | SSD 里可访问的逻辑空间 | 对 RAID 层类似“成员盘容量” |
| Admin Command | 管理命令 | 查询识别信息、健康状态、队列配置 |
| I/O Queue | 读写请求队列 | 和性能、并发、延迟强相关 |
| SMART / Health | 健康状态 | 对遥测、告警、成员盘降级有用 |

## 路线 A：厂商 NVMe/PCIe IP

```text
FPGA RAID/数据面 -> 厂商 NVMe Host IP -> PCIe IP/PHY -> NVMe SSD
```

### 优点

- 最接近工程交付路线；
- 协议栈、PCIe 细节和时序问题由成熟 IP 承担更多；
- 更容易走向真实性能指标和板级验收；
- 有厂商文档、示例工程和支持渠道。

### 代价和风险

- 授权成本高；
- 绑定 FPGA 厂商和器件系列；
- 示例工程可能很重，不适合教程仓库；
- IP 接口、时钟、DMA、缓存、错误处理仍然需要系统集成能力。

### 适合什么时候选

当目标是工程样机、性能验收和真实 SSD 联调时，通常优先评估这条路。

## 路线 B：开源 NVMe Host 参考

```text
FPGA RAID/数据面 -> 开源 NVMe Host -> PCIe core -> NVMe SSD
```

### 优点

- 可以学习协议细节；
- 可读源码，便于研究队列、命令和完成路径；
- 成本可能低。

### 代价和风险

- 成熟度、性能、兼容性不一定满足工程样机；
- 不同 SSD 的行为差异可能放大调试成本；
- PCIe、DMA、cache coherency、错误恢复都可能变成长期任务；
- 许可证、维护状态、仿真环境需要逐项审查。

### 适合什么时候选

适合研究和概念验证；如果要进入工程交付，需要额外做成熟度评估和长时间稳定性测试。

## 路线 C：SoC / CPU 管 NVMe，FPGA 管 RAID 数据面

```text
CPU/SoC/Linux 负责 NVMe 驱动
FPGA 负责 RAID 数据面、校验、缓存或加速
两者通过 AXI/DMA/共享内存协作
```

### 优点

- 复用操作系统 NVMe 驱动；
- 软件调试工具更成熟；
- 健康状态、日志、文件化测试更容易做；
- 适合先把算法和控制策略跑顺。

### 代价和风险

- CPU/FPGA 之间的数据搬运可能成为瓶颈；
- 实时性和确定性不如纯 FPGA 数据通路直观；
- 系统架构会从“纯硬件控制器”变成“软硬协同系统”；
- 需要重新评估 20/40Gbps 指标是否可达。

### 适合什么时候选

适合调研、原型验证、降低 NVMe 接入风险，也适合教学阶段理解系统边界。

## 路线 D：PCIe DMA + 主机模拟盘

```text
FPGA <-> PCIe DMA <-> PC 内存/文件/SSD 模拟服务
```

### 优点

- 不需要一开始接多块真实 NVMe SSD；
- 可以在 PC 侧记录、检查、回放请求；
- 非常适合调试 RAID 映射、错误注入、重建流程；
- 测试可重复，失败更容易定位。

### 代价和风险

- 不是最终工程形态；
- 性能和时序代表性有限；
- PC 侧软件会引入额外变量；
- 不能替代真实 SSD 兼容性和掉电恢复测试。

### 适合什么时候选

适合中期联调和可观测测试平台，不适合作为最终样机路线。

## 路线 E：DDR / BRAM / 文件模型继续做算法验证

```text
RAID controller -> virtual disk port -> BRAM/DDR/file model
```

### 优点

- 最简单、最可控；
- 适合教程和单元测试；
- 能专注 RAID0/1/5、write hole、rebuild、scrub；
- 不会被 PCIe/NVMe 调试淹没。

### 代价和风险

- 离真实 SSD 最远；
- 不能证明真实吞吐；
- 不能覆盖 NVMe 错误、健康状态、队列拥塞；
- 只能作为前期学习和 golden model。

### 适合什么时候选

本仓库第一阶段就选这条路：先把“仓库调度规则”讲清楚，再谈真实高速公路。

## 简单取舍表

| 路线 | 开发量 | 工程风险 | 性能代表性 | 教学友好 | 适合本仓库当前阶段 |
|---|---:|---:|---:|---:|---|
| 厂商 NVMe/PCIe IP | 中到高 | 中 | 高 | 低 | 暂不直接做 |
| 开源 NVMe Host | 高 | 高 | 中到高 | 中 | 只做调研参考 |
| SoC/CPU 管 NVMe | 中 | 中 | 中 | 中到高 | 可作为后续原型路线 |
| PCIe DMA + 主机模拟 | 中 | 中 | 中 | 高 | 可作为后续联调工具 |
| DDR/BRAM/文件模型 | 低 | 低 | 低 | 最高 | 当前首选 |

## 本教程推荐路线

**当前阶段的明确推荐：先做 `abstract disk port`，真实 NVMe Host 只做路线调研，不进入完整实现。**

```text
阶段 0：读需求，拆四个面
阶段 1：Python virtual disk / RAID golden model
阶段 2：RTL 小积木：xor_engine、lba_mapper、后续 stripe_manager
阶段 3：抽象 disk port：读写请求、完成、错误、成员盘状态
阶段 4：再选 NVMe Host 路线：厂商 IP / 开源 / SoC 协同 / DMA 模拟
```

关键原则：

- RAID 层不要直接依赖某个 NVMe IP 的细节；
- 先把成员盘抽象成统一端口；
- 控制面要能表达成员盘健康、错误计数、重建进度；
- 接真实盘之前，先让行为模型和小模块可验证；
- 任何 20/40Gbps 结论都必须在真实接口、时钟、缓存和板级约束下重新验证。

## 进入真实 NVMe 前的评审清单

如果下面问题没有答案，先不要急着把教程升级成真实 NVMe Host 工程；继续停留在 `abstract disk port`、DDR/BRAM 或文件模型阶段更稳。

| 评审项 | 必须回答的问题 | 如果回答不清楚怎么办 |
|---|---|---|
| 板卡与器件 | FPGA 型号是什么？有几个 PCIe 硬核？每个硬核支持几条 lane、什么 Gen？ | 先不要承诺 20/40Gbps，继续用虚拟盘验证 RAID 行为。 |
| IP 与授权 | 是否允许购买厂商/第三方 NVMe/PCIe IP？授权、交付源码、仿真模型、长期维护谁负责？ | 把厂商 IP 作为调研项，不把它写进第一阶段教学闭环。 |
| 性能目标 | 验收到底要求 20Gbps 还是 40Gbps？数据源、包长、条带大小、缓存深度怎么定义？ | 只说“理论目标”，不要声称已满足真实性能。 |
| SSD 与健康状态 | SSD 数量、容量、NVMe 版本、SMART/健康信息、掉盘检测、坏块反馈怎么拿？ | 控制面先保留成员盘状态字段，用模拟错误代替真实盘错误。 |
| RAID/Host 边界 | RAID 层和 Host 层之间的请求、完成、错误、flush、重试语义是否定义清楚？ | 先设计统一 `disk_port`，不要让 RAID 逻辑直接绑定某个 NVMe IP。 |
| 异常与恢复 | 错误注入、掉盘、重建进度、scrub 结果、write hole 风险如何观测？ | 先在 Python 模型和小 RTL demo 中做可重复故障场景。 |
| 稳定性验收 | 72 小时稳定性、温度/功耗/复位/掉电场景、日志留存由谁验证？ | 把这些列入工程验收清单，不放进本仓库第一阶段目标。 |
| 联调环境 | 是否有 PCIe 分析、ILA/VIO、日志采集、可重复数据源和对照脚本？ | 先做 PCIe DMA/主机模拟或纯软件回放平台。 |

一句话决策门槛：

```text
如果 disk_port 的行为模型、错误模型、重建状态都还没跑通，
就不要把风险提前搬到 PCIe/NVMe Host 层。
```

## 给新人一句话

不要把“会做 RAID”和“会接 NVMe SSD”混为一谈。

```text
RAID 是数据怎么分、怎么校验、怎么恢复。
NVMe Host 是怎么安全高效地和真实 SSD 说话。
```

本仓库先教前者，并把后者放进路线图，而不是假装已经解决。
