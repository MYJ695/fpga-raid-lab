# 固存工作模式 - 把工程样机看成一台有状态的机器

## 先抓住这句话

技术要求里的“工作模式”不是装饰词，而是在问：
```text
这台固存样机现在应该做什么？
能不能安全切换？
软件、FPGA、SSD 和测试人员能不能看到同一件事？
```

用费曼法说，固存样机像一台带仓库的录像机：

- 自检模式：开机先检查仓库和货架能不能用；
- 待机模式：暂时不搬货，保持低功耗和可唤醒；
- 记录模式：把外部来的货连续放进仓库；
- 回放模式：把仓库里的货按要求取出来；
- 边记边放模式：一边收新货，一边给别人取旧货。

这个仓库不会做完整模式切换状态机、仲裁器或 NVMe 调度器；它的目标是让你先看懂这些模式为什么会影响架构、接口、验证和验收。

## 五种工作模式一览

| 工作模式 | 人话解释 | 数据面在做什么 | 控制面要看什么 | 验证最关心什么 |
|---|---|---|---|---|
| 自检模式 | 开机体检 | 通常不承载正式载荷数据，可能访问 SSD 做识别和健康检查 | 电源、温度、SSD 健康、阵列状态、错误计数 | 上电后能否在规定时间内进入就绪；坏盘/异常是否能定位 |
| 待机模式 | 暂停搬货但随时可被叫醒 | 数据流量低或停止，缓存和链路保持受控状态 | 低功耗状态、唤醒条件、模式切换请求 | 不应误写数据；唤醒后状态不能乱 |
| 记录模式 | 把输入载荷写入 SSD 阵列 | 多路输入经 AXIS/缓存/RAID 后写向 NVMe SSD | 现在 RAID 模式、条带大小、吞吐统计、错误告警 | 8 路并发输入、持续写入吞吐、掉盘后的降级行为 |
| 回放模式 | 从 SSD 阵列读出历史数据 | 从 NVMe SSD 读出，经 RAID/缓存送往输出侧 | 读地址、读进度、吞吐统计、错误计数 | 读吞吐、数据顺序、坏盘恢复读是否正确 |
| 边记边放模式 | 同时写入和读出 | 写入流、读出流、缓存、NVMe 队列同时竞争资源 | 仲裁状态、拥塞/背压、读写吞吐、告警 | 最容易暴露死锁、缓存不足、调度不公平和状态不同步 |

## 为什么工作模式会影响架构

如果只看 RAID 算法，很容易以为系统只有“读”和“写”。

但工程样机要交付的是一台可测试、可演示、可维护的设备，所以它起码要回答四类问题：

1. **现在处在哪个模式？**  
   软件不能猜，必须能用 AXI-Lite 寄存器或等效控制面读到。

2. **能不能从 A 模式切到 B 模式？**  
   例如记录模式切到待机模式前，是否要等待缓存写空？是否要 flush SSD？

3. **切换失败怎么办？**  
   如果 SSD 不健康、阵列降级、温度异常或链路未就绪，系统应报告原因，而不是静默卡住。

4. **测试人员如何证明它工作过？**  
   需要状态回读、错误计数、事件日志、重建进度、吞吐统计和中断告警等证据。

这就是为什么前期学习不能只看 `xor_engine`。XOR 是 RAID5 的一个积木，工作模式才是系统级运行规则。

## 模式与数据面/控制面的关系

你可以把系统拆成两层：
```text
数据面：真正搬数据的高速通道
控制面：告诉系统现在该做什么，并观察它是否做对
```

| 模式 | 数据面重点 | 控制面重点 |
|---|---|---|
| 自检 | SSD Identify、健康信息读取、阵列可用性检查 | 自检开始/结束、失败原因、健康状态摘要 |
| 待机 | 停止或限制正式数据搬运，保持安全空闲 | 低功耗、唤醒、模式切换许可 |
| 记录 | 输入 AXIS、写缓存、RAID 写路径、NVMe 写队列 | 写入使能、阵列模式、条带参数、吞吐/错误统计 |
| 回放 | NVMe 读队列、RAID 恢复读、输出数据流 | 读任务配置、进度、错误上报 |
| 边记边放 | 读写仲裁、缓存水位、背压、队列调度 | 拥塞状态、优先级策略、死锁/超时告警 |

## 边记边放为什么最难

边记边放不是“记录模式 + 回放模式”简单相加。

它会带来三个额外问题：

1. **带宽分配**  
   写入和读出都想用 NVMe、DDR 缓存和内部总线。谁优先？能不能保证记录不丢？

2. **缓存压力**  
   写入来得太快、读出又占资源时，缓存水位会上升。系统必须能背压、告警或降级处理。

3. **状态一致性**  
   软件看到的是“边记边放中”，硬件内部可能还在 flush、rebuild 或处理错误。控制面必须把关键状态讲清楚。

所以，这个仓库只把边记边放作为架构理解和验收讨论入口，不做完整读写仲裁器。

## 和 RAID 学习的关系

工作模式回答“系统现在在干什么”；RAID 回答“多块盘如何组织数据”。

二者关系可以这样记：
```text
记录模式：最关心 RAID 写路径、write hole、吞吐和掉盘降级。
回放模式：最关心 RAID 读路径、坏盘恢复读和数据顺序。
自检/待机：最关心状态、健康、切换和安全边界。
边记边放：最关心调度、缓存、背压和死锁风险。
```

所以推荐学习顺序是：先看工作模式建立系统状态感，再看 RAID 文档理解数据如何落盘，最后用 Python demo 和 RTL runner 看最小现象。

## 这个仓库能做什么，不能做什么

| 项目 | 这个仓库能做到 | 这个仓库不声称做到 |
|---|---|---|
| 工作模式理解 | 用文档讲清五种模式、数据面/控制面职责、验证关注点 | 完整模式切换 RTL |
| 控制面概念 | 用 `docs/control_plane_registers.md` 和 Python shadow demo 展示状态/IRQ/错误注入思路 | 可综合 AXI-Lite 从设备 |
| 数据面现象 | 用 RAID Python demo 展示映射、校验、write hole、rebuild/scrub | 真实 20/40 Gbps 板级吞吐 |
| 硬件积木 | 用 `xor_engine`、`lba_mapper` runner 展示小模块仿真 | 完整 NVMe RAID 引擎 |

## 读完自测

你可以用下面 5 个问题检查自己是否理解：

1. 自检模式和待机模式为什么不是“什么都不做”？
2. 记录模式下，AXIS、RAID、NVMe、AXI-Lite 分别承担什么角色？
3. 回放模式为什么仍然需要 RAID 恢复读和错误上报？
4. 边记边放为什么容易暴露缓存、仲裁和死锁问题？
5. 为什么这个仓库只讲工作模式框架，而不直接做完整模式状态机？

能答出来，就说明你已经从“会一个 RAID 概念”前进到“能看懂固存样机系统运行框架”。
