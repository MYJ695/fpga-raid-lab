# 需求对齐 - 从《固存样机技术要求》到学习路线

## 核心结论

这份技术要求不是在问“会不会写一个 RAID5 XOR 模块”，而是在问：能不能把 **8 路高速载荷数据**，稳定写进 **多块 NVMe SSD 阵列**，并且在掉盘、重建、监控、验收时都说得清、测得出。

所以本仓库的定位要调整为：

```text
RAID 入门教程
  -> 星载固存样机前期调研
  -> NVMe SSD 阵列的数据面/控制面基础补课
  -> 以后工程实现的知识地图
```

本项目仍然不承诺实现完整工程样机。它只负责把前期问题讲清楚、把最小模型跑起来。

## 文档里的工程目标，用白话说

可以把样机想成一个“太空相机的数据仓库”：

```text
8 路载荷数据流  ->  FPGA 调度与RAID逻辑  ->  4~6块 NVMe SSD
        |                    |                     |
     不停来货             分拣/排队/校验           多个货架并行存放
```

它要做到四件事：

1. **吞吐够高**：8 路输入并发，整机持续写入目标达到 40 Gbps 量级；
2. **阵列可配**：至少 RAID0、RAID1、RAID5，条带大小、重建策略等可配置；
3. **故障可控**：能发现掉盘、隔离故障盘、降级运行、查询重建进度；
4. **交付可验**：接口、寄存器、测试报告、版本关系都要能追溯。

## 关键指标对学习内容的影响

| 技术要求关键词 | 对应要补的基础课 | 本仓库当前处理方式 |
|---|---|---|
| 不少于 4/6 块 NVMe SSD | 多盘条带、成员盘状态、容量计算 | Python RAID0/1/5 模型 + 映射文档 |
| RAID0/1/5 | 条带化、镜像、分布式校验 | 已覆盖概念、demo、基础测试 |
| 8 路输入数据流 | AXIS 流、仲裁、背压、统计 | 先在架构文档建立概念，不深入代码 |
| RAID5 有效写/读 20 Gbps | full-stripe、partial write、RMW、重建读 | 先讲清写路径与 write hole |
| AXI-Lite 管理接口 | 寄存器、状态回读、错误注入 | 先列出寄存器学习清单 |
| AXIS 64-bit 数据接口 | valid/ready、帧边界、字节有效 | 后续 RTL 小实验的接口方向 |
| NVMe 1.2/1.3、PCIe Gen3 x4 | NVMe Host、队列、命名空间、健康日志 | 只做调研入口，不实现协议栈 |
| 遥测与事件 | 温度、SSD健康、阵列状态、错误计数 | 纳入“控制面”知识地图 |
| 尺寸 ≤ 220 mm × 230 mm、总重量 ≤ 3.5 kg、总功耗 ≤ 40 W | SSD 数量、FPGA 板卡、连接器、供电、散热路径和结构空间会互相牵制 | 只提醒这是后续工程约束，不做功耗预算、热仿真、结构设计或安装包络设计 |
| 72小时稳定运行 | 压测、超时恢复、死锁观测 | 当前只解释验收含义，不做板级测试 |

## 本仓库的边界

### 现在做

- 用费曼学习法解释：为什么星载固存需要 RAID/NVMe/FPGA；
- 用 Python demo 看见 RAID0/1/5 的行为；
- 用小 RTL 模块理解 XOR、LBA 映射这类可硬件化积木；
- 把工程样机需求拆成“数据面、控制面、设备面、验证面”。

### 现在不做

- 不实现完整 NVMe Host；
- 不写可上板的 8 路 AXIS 汇聚器；
- 不承诺 40 Gbps/20 Gbps 性能；
- 不做掉电保护、热插拔、完整元数据和板级验收；
- 不把教程伪装成可交付工程代码。

## 读者学完第一阶段应该能回答什么

1. 为什么 RAID0 快但不抗坏盘？
2. 为什么 RAID1 浪费容量但恢复简单？
3. 为什么 RAID5 能坏一块盘还读出数据？
4. 为什么 RAID5 小写比 full-stripe write 麻烦？
5. 为什么工程样机还需要 AXI-Lite 寄存器、状态遥测和错误注入？
6. 为什么 NVMe Host 是独立的大工程，不能和 RAID 入门混在一起一口吃掉？
