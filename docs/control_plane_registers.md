# 控制面寄存器草案 - 先把系统管得住、看得见

## 先抓住这句话

技术要求里反复提到 AXI-Lite、状态回读、故障注入、重建进度、告警和遥测。这里的重点不是“多写几个寄存器”，而是先回答一个工程问题：
```text
软件怎么知道阵列现在能不能写？
如果 SSD 掉线、RAID 降级、重建卡住，软件怎么定位和恢复？
```

换成费曼式说法：
```text
数据面像高速公路，AXIS 负责让车跑起来。
控制面像交通指挥中心，AXI-Lite 负责看路况、改规则、拉警报、做演练。

没有控制面，RAID demo 只能证明“路上有车”；
有了控制面草案，后面才知道工程样机要怎么被软件配置、监控和验收。
```

本页只做前期调研和寄存器地图草案，不做完整 AXI-Lite 从设备，也不承诺板级地址分配已经定稿。

## 原始技术要求在问什么

从《固存样机技术要求》看，控制面至少要覆盖这些点：

| 要求关键词 | 白话解释 | 本教程如何处理 |
|---|---|---|
| AXI-Lite 管理配置接口 | 软件通过 32-bit 寄存器配置和读取状态 | 给出最小寄存器地图草案 |
| 状态回读 | 软件能看到阵列、成员盘、错误计数、吞吐统计 | 把状态拆成阵列、SSD、流、错误四组 |
| 故障注入 | 测试时主动模拟掉盘、超时、校验错误 | 给出错误注入寄存器和安全边界 |
| 重建进度 | RAID5 掉盘后能看到 rebuild 到哪里 | 给出进度、速率限制、控制位 |
| 遥测刷新周期 | 常规监控刷新原则上不大于 1 s，且可配置 | 给出 telemetry interval 配置项 |
| 中断告警 | 关键事件要能提醒软件处理 | 给出 IRQ 状态/屏蔽/清除模型 |
| 版本可追溯 | 硬件、代码、文档和测试报告要能关联 | 给出版本和 build id 寄存器 |

这里最容易漏掉的是：控制面不是验收时才补的“附属功能”。它会反过来影响 RAID 数据通路怎么设计，因为你必须提前想清楚哪些状态要被观测、哪些异常要能被注入、哪些动作必须可停止和可恢复。


## 和技术要求的对应关系

原始技术要求在交付资料里明确要求“接口与寄存器说明”，并要求包含 **地址映射、信号定义及配置流程**。所以这份草案不是随手列寄存器，而是先把后续工程文档要回答的问题提前摆出来：

| 技术要求关注点 | 本文对应设计点 | 后续工程证据 |
|---|---|---|
| 地址映射 | 按 `0x0000`、`0x0100`、`0x0200` 等区间分组 | 最终寄存器表、驱动头文件、测试脚本 |
| 信号定义 | 每个寄存器写清字段含义、读写属性和清除方式 | RTL 接口说明、仿真波形、软件 API |
| 配置流程 | reset -> 设置 RAID/NVMe 参数 -> 使能阵列 -> 观察状态回读 | bring-up 脚本、配置记录、测试报告 |
| 状态回读 | `GLOBAL_STATUS`、成员盘状态、IRQ、错误计数、重建进度 | 运行日志、异常恢复记录、验收截图 |
| 故障注入 | 通过寄存器触发掉盘、读错、写错、超时等场景 | 错误注入用例、恢复时间统计、告警记录 |

一句话记住：寄存器表要能支撑“软件怎么配置、硬件怎么反馈、测试怎么复现”。

## 最小寄存器地图

这不是最终工程地址表，只是学习阶段的可讨论版本。默认每个寄存器 32 bit，地址按 4 字节递增。

| 地址 | 名称 | 方向 | 作用 |
|---:|---|---|---|
| `0x0000` | `VERSION` | RO | RTL/文档版本号，用于追踪交付物一致性 |
| `0x0004` | `BUILD_ID` | RO | 构建编号或 Git 缩写，工程里用于定位 bitstream 来源 |
| `0x0008` | `CAPABILITY` | RO | 支持的 RAID 模式、最大 SSD 数、最大输入流数 |
| `0x0010` | `GLOBAL_CONTROL` | RW | 启动、停止、软复位、进入待机 |
| `0x0014` | `GLOBAL_STATUS` | RO | 未初始化、就绪、运行、降级、重建、故障、待机 |
| `0x0018` | `IRQ_STATUS` | W1C | 中断原因，写 1 清除 |
| `0x001C` | `IRQ_ENABLE` | RW | 中断使能掩码 |
| `0x0020` | `RAID_MODE` | RW | RAID0 / RAID1 / RAID5，通常只允许停止态修改 |
| `0x0024` | `STRIPE_SIZE` | RW | 条带大小配置，后面可从教学 4B 扩展到 512B/4KiB |
| `0x0028` | `MEMBER_MASK` | RW | 现在参与阵列的 SSD 成员掩码 |
| `0x002C` | `HOT_SPARE_MASK` | RW | 预留热备盘掩码，学习阶段可只保留概念 |
| `0x0030` | `REBUILD_CONTROL` | RW | 开始、暂停、继续、取消重建 |
| `0x0034` | `REBUILD_STATUS` | RO | 空闲、扫描中、写入中、完成、失败 |
| `0x0038` | `REBUILD_PROGRESS` | RO | 重建百分比或完成条带数 |
| `0x003C` | `REBUILD_THROTTLE` | RW | 重建限速，避免抢占正常写入带宽 |
| `0x0040` | `SCRUB_CONTROL` | RW | 启动/停止一致性检查 |
| `0x0044` | `SCRUB_STATUS` | RO | scrub 状态、最近 mismatch 标志 |
| `0x0048` | `TELEMETRY_INTERVAL` | RW | 遥测刷新周期，工程目标可按 <=1 s 约束设计 |
| `0x0050` | `ERROR_INJECT_CONTROL` | RW | 故障注入总开关和类型选择 |
| `0x0054` | `ERROR_INJECT_TARGET` | RW | 选择 SSD、输入流、LBA/stripe 或命令队列 |
| `0x0058` | `ERROR_INJECT_PARAM` | RW | 注入次数、持续时间、错误码等参数 |
| `0x0060` | `ERROR_SUMMARY` | RO | 全局错误摘要 |
| `0x0064` | `ERROR_COUNTER_CLEAR` | WO | 清除统计计数器 |

更真实的工程会继续细分地址空间。例如：每个输入流一组统计，每个 SSD 一组状态，每个 RAID 子模块一组调试计数。学习阶段先把“有哪些组、为什么要有”讲清楚，比直接堆完整地址表更有用。

## 状态机先不要复杂，先能讲清楚

控制面最小状态机可以先画成这样：
```text
RESET
  -> SELF_TEST
  -> READY
  -> RUNNING
  -> DEGRADED
  -> REBUILDING
  -> READY/RUNNING

任意关键故障
  -> FAULT

软件请求低功耗
  -> STANDBY
```

每个状态必须回答三个问题：

1. 这个状态下允许哪些寄存器写入？
2. 这个状态下数据面能不能继续接收 8 路输入？
3. 如果发生掉盘、超时、复位或异常下电，下一步怎么恢复？

例如，`RAID_MODE` 不应该在 `RUNNING` 状态随便改。否则软件把 RAID5 切到 RAID0，数据保留策略会立刻说不清。技术要求里也明确提到，模式切换应在阵列停止或重新初始化后执行，并明确数据保留、清空和重新构阵策略。

## 成员盘状态怎么设计

成员盘不是只有“好/坏”两种状态。起码要区分：

| 状态 | 含义 | 软件动作 |
|---|---|---|
| `ABSENT` | 没识别到盘 | 不参与阵列，提示检查连接或供电 |
| `PRESENT` | 物理存在，但未加入阵列 | 等待初始化或人工确认 |
| `ACTIVE` | 正常参与读写 | 正常统计吞吐、错误和健康状态 |
| `DEGRADED` | 阵列降级运行 | 限制某些操作，提醒尽快重建 |
| `FAILED` | 读写失败或超时达到阈值 | 隔离该盘，触发告警 |
| `REBUILDING` | 正在被重建 | 更新进度，允许限速 |
| `SPARE` | 热备盘 | 等待替换故障成员 |
| `MASKED` | 被软件屏蔽 | 用于测试或维护，不自动参与阵列 |

这张表的作用是避免后面把所有异常都塞进一个 `disk_error` 位。一个 bit 很容易写，但验收时解释不清：到底是没插盘、掉盘、健康告警、命令超时，还是人工屏蔽？

## 错误注入要有边界

错误注入不是为了把系统搞坏，而是为了验证“坏了以后能不能被看见、被隔离、被恢复”。

建议第一阶段只定义这些注入类型：

| 注入类型 | 目标 | 想验证什么 |
|---|---|---|
| 掉盘 | SSD 成员 | RAID1/RAID5 是否进入降级，告警是否产生 |
| 读超时 | SSD 或命令队列 | 软件能否看到 timeout counter 和错误码 |
| 写失败 | SSD 成员 | RAID 写路径能否停止或降级处理 |
| parity mismatch | RAID5 stripe | scrub 是否能发现一致性问题 |
| 输入流暂停 | AXIS 输入流 | valid/ready 背压和流量统计是否合理 |

边界也要写清楚：

- 错误注入默认必须关闭；
- 只有测试模式下允许打开；
- 注入目标必须可回读；
- 注入后产生的错误计数和事件记录不能悄悄丢失；
- 工程样机里还要考虑权限、误触发保护和恢复流程。

## 重建和 scrub 为什么要分开

这两个词容易混。
```text
rebuild = 已经有盘坏了，把丢失的数据重新算出来，写到新盘或热备盘。
scrub   = 平时后台检查数据和 parity 是否一致，尽早发现潜在错误。
```

RAID5 最怕的是：已经降级了，又在重建前发现另一个成员盘有隐藏错误。所以工程里会关心一致性检查、错误计数、重建进度和重建限速。

学习阶段不需要做完整 rebuild 控制器，但控制面要提前留出这些问题：

1. 重建从哪个 LBA/stripe 开始？
2. 现在完成了多少？
3. 失败原因是什么？
4. 是否允许暂停、继续、取消？
5. 重建会不会影响正常 20/40Gbps 指标？
6. scrub 发现 mismatch 后，是只告警、自动修复，还是等待软件确认？


## 最小软件流程：从上电到异常恢复

把寄存器表换成软件视角，可以按下面这条线理解：
```text
1. reset 后读取 VERSION / BUILD_ID / CAPABILITY，确认软硬件版本匹配。
2. 写 IRQ_ENABLE，打开掉盘、错误注入完成、rebuild 完成、scrub mismatch 告警。
3. 停止态写 RAID_MODE、STRIPE_SIZE、MEMBER_MASK。
4. 写 GLOBAL_CONTROL.enable，让阵列进入 RUNNING。
5. 定期读 GLOBAL_STATUS、成员盘状态、错误计数和 TELEMETRY_INTERVAL_MS。
6. 测试模式下写 ERROR_INJECT_CONTROL，触发掉盘或 parity mismatch。
7. 发现 IRQ_STATUS 后，软件读取状态、记录日志，再用 W1C 写 1 清除已处理 bit。
8. 对 FAILED 成员启动 REBUILD_CONTROL，轮询 REBUILD_PROGRESS，完成后确认成员回到 ACTIVE。
9. 后台运行 SCRUB_CONTROL，若 SCRUB_MISMATCH_COUNT 增加，进入一致性问题处理流程。
```

这条流程不是驱动代码，而是告诉读者：控制面不是一张静态表，它是一套“配置 -> 观察 -> 注入 -> 恢复 -> 留证据”的闭环。

## 最小控制面验收用例

| 场景 | 软件动作 | 期望状态 | 可留下的证据 |
|---|---|---|---|
| bring-up | 配置 RAID5、成员盘掩码、IRQ_ENABLE，再 enable | `GLOBAL_STATUS=RUNNING`，成员盘为 `ACTIVE` | 配置日志、寄存器 dump |
| 掉盘注入 | 写 `ERROR_INJECT_CONTROL=FAIL_MEMBER_2` | SSD#2 变 `FAILED`，阵列 `DEGRADED`，`IRQ_STATUS` 置位 | 注入命令、IRQ 日志、成员状态截图 |
| IRQ 清除 | 软件处理后对 `IRQ_STATUS` 写 1 清除 | 已处理 bit 清零，未处理 bit 保留 | W1C 操作记录 |
| rebuild | 写 `REBUILD_CONTROL=START_MEMBER_2` | `REBUILD_PROGRESS` 到 100，成员回 `ACTIVE` | 进度日志、完成事件 |
| scrub mismatch | 运行 scrub 并注入 parity mismatch | `SCRUB_MISMATCH_COUNT` 增加，告警置位 | scrub 日志、错误计数 |

可以运行 `labs/level0_python_model/demo_control_plane.py` 看这些状态如何变化。

## 第一阶段的学习目标

读完这一页，应该能回答：

1. 为什么 AXI-Lite 不是“低速可有可无接口”，而是工程验收的入口？
2. 为什么 RAID demo 需要版本号、状态回读、错误计数和告警？
3. 为什么成员盘状态不能只有 good/bad？
4. 为什么错误注入必须配套事件记录和清除机制？
5. 为什么 rebuild 和 scrub 要分开设计？
6. 为什么寄存器表要和文档、测试报告、bitstream 版本一起可追溯？

## 这个仓库现在先不做什么

- 不做完整 AXI-Lite 从设备；
- 不承诺最终工程地址映射；
- 不做完整 RAID rebuild/scrub 控制器；
- 不连接真实 NVMe SSD；
- 不把寄存器草案说成已经用板级验收。

这个仓库已经给出一个简单探索：`labs/level0_python_model/demo_control_plane.py`。它模拟软件写 `RAID_MODE`、`IRQ_ENABLE`、`ERROR_INJECT_CONTROL`，再观察 `GLOBAL_STATUS`、`IRQ_STATUS`、重建进度和 scrub mismatch 计数如何变化。这个实验能帮助理解控制面，但仍然不是工程实现。
