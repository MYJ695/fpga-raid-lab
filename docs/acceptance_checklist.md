# 验收清单 - 把技术要求翻译成可检查的问题

## 核心结论

本仓库不是工程样机交付物，所以这里的“验收清单”不是板级验收报告。

它的作用是把《固存样机技术要求》里的验收压力翻译成三类问题：

```text
1. 本教程已经能帮你理解什么？
2. 本教程的小实验最多能证明什么？
3. 真正工程交付还必须补哪些证据？
```

用费曼法说：

```text
跑通 demo = 你会在沙盘上解释调度规则。
工程验收 = 真实仓库在高峰期、坏货架、停电、连续值班时仍能交付记录。
```

所以，Python/RTL 小实验是学习闭环，不是 20/40Gbps 样机证明。

## 1. 功能验收视角

| 技术要求里的问题 | 本仓库帮你理解什么 | 当前小实验能证明什么 | 后续工程还要补什么 |
|---|---|---|---|
| RAID0 | LBA 如何条带化到多块盘 | Python RAID0 映射、RTL `lba_mapper` 基础映射 | 多盘并发调度、真实 SSD 写入一致性、异常恢复 |
| RAID1 | 镜像为什么牺牲容量换恢复简单 | Python 镜像读写行为 | 写双盘一致性、坏盘隔离、恢复策略 |
| RAID5 | XOR parity 如何支持单盘故障恢复 | Python full-stripe、degraded read、scrub/rebuild demo；RTL `xor_engine` | partial write、元数据、掉电保护、重建限速、长期一致性 |
| 条带/成员盘映射 | 逻辑地址和成员盘位置的关系 | 小规模 LBA/chunk 演示 | 512B/4KiB 块、真实条带大小、容量边界、磨损和坏块策略 |
| 错误注入 | 为什么验收要能主动制造掉盘/读错/校验错 | 文档和 Python demo 展示坏盘、write hole、scrub mismatch | AXI-Lite 寄存器、错误码、日志、告警、中断、恢复脚本 |
| 重建与 scrub | 坏盘后不是“找回原盘”，而是用幸存数据重算 | demo 展示 rebuild/scrub 的基本现象 | 后台重建、限速、进度查询、失败重试、重建中继续读写 |

## 2. 性能验收视角

| 指标 | 本仓库现在怎么处理 | 不能误解成什么 | 后续必须补的证据 |
|---|---|---|---|
| 8 路 AXIS 64-bit 输入 | 用 `docs/axis_axi_lite_basics.md` 解释 valid/ready、背压、仲裁 | 不能证明已经实现 8 路高速汇聚 | stream ingress RTL、缓存深度分析、仲裁策略、压力测试 |
| RAID5 有效写/读 20Gbps | 用文档解释 full-stripe、RMW、reconstruct write 和性能瓶颈 | 不能证明 Python/小 RTL 能跑到 20Gbps | 时钟频率、总线宽度、流水线、DDR/BRAM 缓冲、真实 SSD 写入曲线 |
| 整机 40Gbps 量级写入 | 在需求地图中标成工程目标 | 不能用单模块仿真替代整机吞吐 | 板级 PCIe/NVMe/AXIS 联调、端到端吞吐日志、长时间压力曲线 |
| 多块 NVMe SSD | 用 `docs/nvme_host_options.md` 说明 Host/IP/SoC/DMA 路线 | 不能假装教程已经有 NVMe Host | IP 选型、PCIe lane/Gen、队列深度、SSD 型号、温度降速测试 |

关键提醒：

```text
吞吐 = 接口宽度 × 时钟 × 有效利用率 - 仲裁/缓存/重建/错误处理开销
```

只看一个 XOR 模块或一个 LBA 映射模块，不能推出整机 20/40Gbps。

## 3. 稳定性和异常恢复视角

| 验收关注点 | 本仓库能做的前期准备 | 工程样机必须补的真实验证 |
|---|---|---|
| 72 小时稳定运行 | 解释为什么要看死锁、超时、错误计数和日志 | 真实板卡连续压测、温度/电源/复位记录、失败自动定位 |
| 掉盘/坏盘 | Python 模型展示 degraded read 和 rebuild | 真实 SSD 掉线、复位、重新枚举、阵列降级策略 |
| 掉电/write hole | 文档解释 data/parity 不一致风险 | 掉电保护、电容/日志/元数据、恢复扫描流程 |
| 温度和健康状态 | 控制面地图中列入遥测主题 | SMART/Health log 获取、温度降速、寿命告警 |
| 复位和重启 | 当前只作为验收问题提示 | 寄存器默认值、阵列元数据加载、未完成写入恢复 |

## 4. 控制面和可观测性视角

验收不是只问“数据有没有写进去”，还会问：系统坏的时候能不能说清楚。

最低限度要能回答：

1. 当前 RAID 模式是什么？RAID0/RAID1/RAID5 如何切换和锁定？
2. 每块成员盘状态是什么？online、failed、rebuilding、spare、missing 如何表示？
3. 错误计数如何读出？读错、写错、校验错、超时是否分开统计？
4. 重建进度如何观测？百分比、当前 LBA、剩余时间、失败原因在哪里？
5. 错误注入如何触发？触发后是否能清除、是否留日志？
6. 版本信息如何追溯？RTL、软件、测试脚本、需求版本是否对应？

本仓库当前只完成概念入口；后续可用 `docs/control_plane_registers.md` 把这些问题变成寄存器草案。

## 5. 资料交付视角

真实工程交付时，建议至少形成下面这条证据链：

```text
技术要求条款
  -> 架构设计说明
  -> 模块设计说明
  -> 测试用例
  -> 测试日志/波形/截图
  -> 问题单和版本记录
```

本仓库当前能提供的是前两层的学习材料和少量小实验测试，不替代工程测试报告。

## 6. 学完本仓库后，你应该能回答什么

如果下面问题答不上来，就不要急着进入真实 NVMe/PCIe 工程：

1. 为什么 RAID0 快，但任何一块盘坏了都可能丢数据？
2. 为什么 RAID1 容量利用率低，但恢复路径简单？
3. RAID5 的 parity 是怎么由数据块 XOR 出来的？
4. RAID5 为什么最怕 partial write 被打断？
5. rebuild 和 scrub 的区别是什么？
6. 8 路 AXIS 输入为什么需要仲裁、缓存、背压和统计？
7. AXI-Lite 寄存器为什么是验收调试的入口？
8. NVMe Host 为什么不是“多写几行 RAID RTL”就能完成？
9. 为什么 abstract disk port 是第一阶段推荐路线？
10. 为什么跑通 Python/RTL 小实验仍不能证明 20/40Gbps 和 72 小时稳定性？

## 7. 是否可以进入下一阶段？

| 判断问题 | 可以进入真实工程选型的信号 | 应继续停留在教程/抽象盘阶段的信号 |
|---|---|---|
| 需求 | RAID 模式、盘数、吞吐、稳定性、验收方式已经明确 | 只知道“要做 NVMe RAID”，指标和测试方式不清楚 |
| 板卡 | FPGA、PCIe hard block、lane/Gen、时钟资源已确认 | 板卡/IP/SSD 都未定 |
| 数据面 | 8 路输入速率、帧格式、背压策略已明确 | 只停留在“有 8 路数据” |
| 存储面 | NVMe Host 路线、IP 授权、SSD 型号和测试工具已明确 | 还不知道自研、采购 IP、SoC 协同还是 DMA 模拟 |
| 验证面 | 有端到端压力测试和异常注入计划 | 只有单元 demo，没有系统级验收计划 |

结论：

```text
本仓库的任务，是让你有资格提出正确的工程问题；
真正的样机验收，还需要板级实现、接口联调和长时间测试证据。
```
